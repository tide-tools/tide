# tide-routines

Рутина (`kind: routine`) — это переиспользуемая процедура, оформленная как арка-контейнер.
Сначала прочитай [[tide-flow]]: «всё есть арка» и «работа ведётся ВНУТРИ арки
(`workspace/`)» — рутины этого не отменяют, а наследуют.

## Модель (держи её в голове)

```
рутина (контейнер, kind: routine)        ← переиспользуемый ШАБЛОН
  ## steps        — процедура (runbook): как делать
  ## experience   — что накопилось по ранам (уроки, гочи)
  arcs/
    01-<run>      ← КЕЙС применения = РАН = вложенная арка (сессия)
    02-<run>      ← каждый ран может сам разрастись в ветку
    ...
```

- **Шаблон ≠ исполнение.** Шаблон (steps/experience) живёт на КОНТЕЙНЕРЕ рутины.
  Конкретное применение — отдельный **ран** (вложенная арка), у него свои
  input/workspace/output.
- **Жизненный путь:** есть арка, в ней повелась работа → ты понимаешь, что это
  повторяемо → **оформляешь в рутину** (выносишь процедуру в `## steps`). Дальше
  каждый раз «надо опять сделать X» = новый ран, а не новая арка с нуля и не дубль
  рутины.
- **Раны нумеруются и цепляются** внутри рутины (`01,02,…`), как сессии в треде.
- **Опыт копится:** после рана допиши в `## experience` рутины, что он научил —
  чтобы следующий ран был умнее.

## Команды (CLI — твой API)

```bash
# 1. Завести рутину (контейнер)
tide arc new-routine <slug>
#   → .tide/arcs/NN-@<slug>/  с паспортом <slug>-goal.md (kind: routine)
#   Заполни ## steps (процедуру) и шапку — НЕ оставляй плейсхолдеры <…>.

# 2. Завести РАН рутины (вложенная арка-сессия)
tide arc new-session <run-slug> -p <routine-slug>
#   → .tide/arcs/NN-@<routine>/arcs/MM-<run-slug>/
#   -p принимает slug рутины так же, как slug треды.

# 3. Прогон: веди ран ВНУТРИ его арки
#   все файлы/артефакты/логи прогона → <run>/workspace/  (см. tide-flow, правило 2)
#   следуй ## steps рутины; сверяйся с ## experience.

# 4. Закрыть ран + записать опыт
#   durable-итог рана → <run>/output/ ; затем допиши урок в ## experience рутины ;
#   tide arc close <run-slug>
```

При запуске рана через `tide menu` (Type → Routine) сид автоматически подкладывает
процедуру рутины (`## steps`/`## experience`) — это и есть твой runbook на прогон.

## Как НЕ надо (типовые перекосы)

- ❌ **Раны болтаются отдельными арками верхнего уровня** вместо вложения в рутину.
  Тогда рутина не копит свои кейсы, а дерево засоряется. Ран всегда `-p <routine>`.
- ❌ **Дубль рутины** на то, что уже есть. Перед `new-routine` глянь `tide status` —
  если живая рутина того же смысла уже есть, заводи РАН в неё, не плоди новую.
- ❌ **Пустой шаблон в active.** Не оставляй `goal`/`## steps` плейсхолдерами `<…>` —
  пустой контейнер неотличим от живого и мусорит борд.
- ❌ **Работа рана мимо `workspace/`** (в /tmp, в корень). Всё по [[tide-flow]].

## Паттерны

- **Рутина на выпуск кодов** — `issue/verify/list/revoke` как `## steps`; каждый батд =
  ран; `## experience` хранит «что сломалось в прошлый раз». (Пример: mitehq
  `15-@invite-codes`.)
- **Рутина на деплой** — обычно ОТДЕЛЬНЫЕ рутины на стенд и на прод (разные хосты/
  гейты/риск), не один шаблон с флагом. Каждый деплой = ран; `## experience` несёт
  посмертия прошлых выкаток.

## Скилл сломался или затупил? Зарепорти

Если рутинный CLI/модель повели себя криво или инструкция не сошлась с реальностью —
урони репорт в инбокс `tide-stack`, не чини молча:

```bash
tide candidate add skill-routines-<кратко> \
  "tide-routines: что делал → ожидал → произошло (точная команда/ошибка). Воспроизведение." \
  --project tide-stack
```

Ляжет в бэклог `tide-stack` с `from: ↗ <твой проект>` — человек разберёт и починит.
