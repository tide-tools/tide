# tide-flow

Ты в сессии, поднятой tide (`tide menu` → проект → треда/рутина → сессия). Сид уже
дал тебе CANON.md проекта, паспорт активной арки и ростер. Этот скилл — про то, как
здесь **вести работу**, чтобы она не растекалась.

CLI — это твой API. Команды детерминированы; `tide help` и `tide <cmd> --help` —
источник истины по флагам. Скилл даёт МОДЕЛЬ, не дублирует справку.

## Два железных правила

### 1. Всё есть арка
Любая единица работы живёт как арка (`NN-<slug>/` в `.tide/arcs/`). Тред
(треда = `kind: thread`) — контейнер сессий одной линии работы; рутина
(`kind: routine`) — контейнер ранов переиспользуемой процедуры (см. [[tide-routines]]).
Не заводи «работу мимо арки»: нет подходящей — заведи (`tide arc new <slug>` /
`tide arc new-session <slug> -p <thread>`), а не пиши в воздух.

### 2. Работа ВЕДЁТСЯ ВНУТРИ арки
Это ядро ценности tide: по арке всегда видно, что и как делалось. Жёсткая семантика
трёх папок арки:

| Папка | Что туда | Когда |
|-------|----------|-------|
| `input/` | с чего арка началась — сид, ТЗ, исходники | при заходе |
| `workspace/` | **ВСЯ текущая работа** — промежуточные файлы, скачанное, обработанное, логи прогонов, черновики, заметки по ходу | пока работаешь |
| `output/` | чем арка ЗАКОНЧИЛАСЬ — durable-пойнтеры только на финал | при закрытии |

Конкретно: если задача идёт во внешний сервис, грузит картинку, обрабатывает её,
зовёт API, генерит артефакт — **всё это кладётся в `<arc>/workspace/`**, а не в `/tmp`,
не в корень проекта, не «рядом». Прогресс/события/решения по ходу — тоже в workspace
(заметкой/логом). `output/` трогаешь только когда арка завершается.

Не уверен, в какой ты арке → `tide status` (борд) покажет открытые арки; активную
тебе назвал сид. Путь арки = `.tide/arcs/<NN-slug>/` (для сессии —
`.tide/arcs/<NN-@thread>/arcs/<NN-slug>/`).

## Цикл работы

1. **Сориентируйся** — `tide status`. Прочитай паспорт активной арки (его `## cursor`
   — точка возобновления). Не начинай, пока не понял, где работа.
2. **Веди в арке** — все файлы/прогресс в `workspace/` (правило 2). По ходу обновляй
   `## cursor`/`## context` в паспорте, чтобы следующая сессия подхватила.
3. **Заметил постороннее** — урони кандидатом, не отвлекаясь (см. ниже).
4. **Закрой** — когда сделано: положи durable-итог в `output/`, затем
   `tide arc close <slug>` (гейт не даст закрыть с пустым output / плейсхолдерами).

## Кандидаты — не теряй идеи

Всплыло будущее-дело, которое НЕ относится к текущей арке — не делай его сейчас,
урони в бэклог:

```bash
tide candidate add <slug> "<полная идея>"            # в ТЕКУЩИЙ проект
tide candidate add <slug> "<идея>" --project <name>  # в СОСЕДНИЙ проект из ростера
```

Кросс-проектный вариант: работаешь в проекте A, заметил, что что-то относится к
проекту B (он есть в ростере — см. блок Roster в сиде) → кидаешь кандидата прямо в B,
он ляжет в его бэклог с пометкой `from: ↗ A`. Не переключай контекст, не тащи чужое в
свою арку. Захват дешёвый; промоутит кандидата в арку только оркестратор того проекта
(`tide candidate promote`), локально.

## Роли (коротко)
- **orchestrator** — ведёт CLI: открывает/закрывает арки, мёрджит канон, промоутит
  кандидатов. Команды знаешь ты, не человек.
- **worker** — работает ОДНУ открытую арку; пишет только в её `workspace/`/`output/` +
  свой кандидат-дроп. Канон не мёрджит, кандидатов не промоутит.

Твою роль назвал сид (и `TIDE_ROLE`).

## Скилл сломался или затупил? Зарепорти (важно)

tide-скиллы (`tide-*`) — живой инструмент, и ты их главный пользователь. Если скилл
из этой группы повёл себя криво, команда упала, инструкция не сошлась с реальностью,
или ты затупил из-за неясной формулировки — **не молчи и не чини молча**. Урони репорт
в проект-инбокс `tide-stack` одной командой:

```bash
tide candidate add skill-<имя>-<кратко> \
  "Скилл <имя> v<если знаешь>: что делал → что ожидал → что произошло (точная ошибка/команда). Как воспроизвести." \
  --project tide-stack
```

Ляжет в бэклог `tide-stack` с пометкой `from: ↗ <твой проект>` — человек придёт,
разберёт и починит скилл. Репортить дёшево; это и есть «история проблем» скилла
(бэклог + git самого скилла). Чем точнее воспроизведение — тем быстрее фикс.
